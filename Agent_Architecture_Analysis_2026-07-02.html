1. AGENT-ARCHITEKTUR: Hybrid (Decision-Trees + Fuzzy Logic + Machine Learning)

1.1 Decision-Trees (schnelle, strukturierte Entscheidungen)

Einsatz: Erste Ebene — Agent klassifiziert schnell und deterministisch

Eingabe: Marktdaten (BTC-Trend, Dominanz, Fear&Greed, Asset-Daten) ↓ [1] Regime-Baum (BTC-Trend × Dominanz) → BULLE / BÄR / SEITWÄRTS / EXTREM ↓ [2] Asset-Klassifikation (Marktkap, Alter, Volatilität) → Tier 1 / 2 / 3 / SUSPENDED ↓ [3] Regel-Chain (Confluence: EMA + RSI + Volumen) → SIGNAL JA / NEIN / UNSICHER

Eigenschaften:

  • ✓ Deterministisch (immer gleiche Inputs → gleiche Outputs)
  • ✓ Schnell (Millisekunden)
  • ✓ Billig (keine Claude-Calls)
  • ✓ Transparent (Z-4 Nachvollziehbarkeit)
  • ✓ Lokal läufig — Agent funktioniert ohne Internet/Claude

1.2 Fuzzy Logic / Scoring (kontinuierliche Bewertung)

Einsatz: Zweite Ebene — aus Decision-Tree-Ausgaben wird ein kontinuierlicher Score

Decision-Tree sagt: "Signal JA" (binär) Fuzzy-Scoring fügt Feinheiten hinzu: - Confluence-Strength: wie viele Indikatoren stimmen überein? (0-100%) - Regime-Fitness: passt das Signal zur aktuellen Marktphase? (0-100%) - Asset-Risk: wie riskant ist dieses Asset? (0-100%) - Source-Reliability: Zuverlässigkeit der Datenquellen? (0-100%) → FINALE SIGNAL-KONFIDENZ = gewichtete Kombination (0-100%)

Eigenschaften:

  • ✓ Nuanciert (kein All-or-Nothing)
  • ✓ Gewichtet (wichtige Faktoren zählen mehr)
  • ✓ Lernbar (Gewichte können sich ändern)
  • ✓ Auch lokal (keine Claude-Abhängigkeit)

1.3 Machine Learning lokal (Agent trainiert sich selbst)

Einsatz: Dritte Ebene — Agent verbessert seine Entscheidungen über Zeit

[Phase 1: Backtesting] Agent: historische Signale + Kurse + tatsächliche Outcomes → "War das Signal gut oder schlecht?" [Phase 2: Fehler-Analyse] Agent erkennt: "Im Bärenmarkt unterschätze ich Risiken" "Bei Small Caps falsch kalibriert" "Quelle XYZ hat 30% False Positives" [Phase 3: Parameter-Tuning] Agent passt automatisch an: - Fuzzy-Gewichte (Fundamental: 40% → 50% im Bärenmarkt) - Schwellenwerte (RSI: 70 → 65 für Tier-3) - Quellen-Zuverlässigkeit (Quelle XYZ: Malus +20%) [Phase 4: Validierung] Agent testet neue Parameter Wenn besser → adoptieren; wenn schlechter → revertieren

Technologie-Optionen (lokal, kein Claude):

  • Option A: Einfache Regression (sklearn) — schnell, interpretierbar
  • Option B: Decision-Tree-Learning (XGBoost/LightGBM) — robust, auto-tuning
  • Option C: Bayesian Optimization (hyperopt) — findet optimal Schwellenwerte

Eigenschaften:

  • ✓ Selbst-verbessernd (Agent lernt aus Fehlern)
  • ✓ Lokal (SQLite + Python ML, keine Cloud)
  • ✓ Transparent (Gewichte + Tuning dokumentiert)
  • ✓ Evolutionär (Parameter ändern langsam, dokumentiert)

2. CLAUDE-ROLLE: Zusätzlich, nicht zentral

Claude kommt ins Spiel für 3 spezialisierte Aufgaben — nicht für everyday Signaling

2.1 Meta-Intelligenz & strategische Überprüfung

Wann: Agent triggert Claude bei Unsicherheiten

Beispiel: Agent sagt "Meine Signale im Bärenmarkt haben 40% Win-Rate. Makro-Situation ungewöhnlich (Zentralbank-Pivot). Sollte ich Gewichte ändern?"

Claude liest Makro-Context, denkt strategisch, sagt ja/nein mit Begründung.

Frequenz: Ad-hoc, ~2-5× pro Woche (nicht täglich)

2.2 Proaktive Gesamt-Analysen (2× täglich)

Wann: 2× täglich (z.B. 06:00 & 18:00 UTC)

Claude liest: Agent-Signale, Makro-News, Fear&Greed, BTC-Dominanz. Denkt: "Was passt nicht zusammen? Versteckte Risiken?"

Claude schreibt: Kurz-Memo an Nutzer + Agent

Cost-Profile: 2 API-Calls/Tag = ~60/Monat (geplant)

2.3 User-Dialog & Lernprozess-Überprüfung

Wann: Nutzer fragt Claude oder Agent bittet um Learning-Review

Cost-Profile: On-demand (Nutzer-getriggert)

3. DB-ARCHITEKTUR: Shared Source of Truth

Wichtig: DB ist NICHT nur Cache, sondern die Koordinations-Ebene für alle 3 Komponenten (Agent, GUI, Claude)

3.1 Tabellen-Struktur

[CORE] - watchlist (symbol, name, typ, status, coingecko_id, ...) - bestände (symbol, anzahl, kaufpreis, ...) - preise (symbol, timestamp, kurs, volumen, ...) [AGENT-OUTPUT] - signale (symbol, timestamp, aktion, kurzbegründung, konfidenz, ...) - regime_state (timestamp, regime, btc_trend, dominanz, fear_greed, ...) - indikatoren (symbol, timestamp, ema_20, ema_50, rsi, ...) - kandidaten (symbol, timestamp, quelle, begründung, status, ...) [LEARNING & OPTIMIZATION] - signal_outcomes (signal_id, tatsächlicher_kurs, gewinn_verlust, war_gut, ...) - agent_parameter_history (name, alter_wert, neuer_wert, timestamp, warum, ...) - fehler_muster (asset_klasse, regime, fehlertyp, häufigkeit, ...) [CLAUDE-INTEGRATION] - claude_calls (timestamp, triggerer, input, output, tokens_used, kosten, ...) - claude_memos (timestamp, typ, inhalt, action_taken, ...) [USER-CHANGES] - user_overrides (timestamp, bereich, alter_wert, neuer_wert, grund, ...) [AUDIT] - audit_log (timestamp, komponente, aktion, status, fehler, ...)

3.2 Synchronisations-Logik

Agent-Zyklus (kontinuierlich, alle 15 Min):

  1. Agent liest: watchlist, bestände, regime_state, preise (aus DB)
  2. Agent berechnet: neue Indikatoren, Signale, Regime-Update → RAM
  3. Agent schreibt: Signale, Regime-Changes, Kandidaten → DB (mit LOCK)
  4. Agent loggt: Audit-Trail → audit_log

GUI-Zyklus (kontinuierlich, wenn Nutzer die App offen hat):

  1. GUI liest: watchlist, bestände, signale, regime_state (aus DB)
  2. GUI zeigt: aktuellen Stand an
  3. Nutzer ändert: Position, Override, Quelle → GUI schreibt in DB (mit LOCK)
  4. GUI liest: user_overrides + agent_parameter_history → zeigt Änderungen

Claude-Zyklus (triggered + 2× täglich):

  1. Claude liest: signale, signal_outcomes, regime_state, Makro-News (DB + extern)
  2. Claude denkt: Meta-Analyse
  3. Claude schreibt: Memo, ggf. Empfehlungen → claude_memos (mit LOCK)
  4. Claude loggt: tokens_used → claude_calls (für Cost-Tracking)

Learning-Zyklus (täglich kurz + wöchentlich tiefgehend):

  1. Agent liest: signal_outcomes (was ist mit meinen Signalen passiert?)
  2. Agent analysiert: Fehler-Muster
  3. Agent schreibt: neue Parameter → agent_parameter_history
  4. Agent passt automatisch an oder schlägt Claude vor

4. TECHNISCHE KONSEQUENZEN

4.1 Performance & Latenz

Komponente Latenz Frequenz Last
Agent-Indikator-Zyklus <100ms alle 15 Min mittel
Agent-Backtest 1-5 sec tägl. kurz / wöchentl. tiefgehend hoch (asynchron)
Claude-Call (strategisch) 2-5 sec ad-hoc, 2x tägl. Meta niedrig-mittel
GUI-Read <100ms kontinuierlich niedrig
GUI-Write (Override) <100ms ad-hoc niedrig

Bottleneck: Claude-Calls (5 sec) — aber nicht kritisch, da nicht für Echtzeit-Signale

4.2 Concurrency & DB-Locking

Problem: Agent + GUI + Claude schreiben gleichzeitig in DB

Lösung (SQLite-gerecht):

  • Write-Locks pro Tabellen-Bereich (Agent sperrt signal-Tabelle, wenn schreibt)
  • Queue-Pattern: Änderungen sammeln, batch-verarbeiten
  • Transaktionen: BEGIN → Änderung → COMMIT mit Timeout-Handling
Konsequenz: Manchmal wartet eine Komponente kurz — aber OK, nicht häufig

4.3 Claude-Kosten & Budget-Tracking

Claude-Nutzung:

  • Ad-hoc strategische Überprüfungen: ~2-5 Calls/Woche (Agent unsicher)
  • 2× tägl. Gesamt-Analysen: 2 Calls/Tag = ~60 Calls/Monat
  • User-Dialog: variable (Nutzer-getriggert)

Geschätzter Verbrauch: ~100-200 API-Calls/Monat (mit 3x-Puffer ~500)

Cost-Tracking in GUI:

[Dashboard] Claude Budget: $10/Monat (z.B.) Bereits verbraucht: $2,40 (24%) Letzte 30 Tage: 120 Calls Diese Woche: 18 Calls

Phase 2 Learning (ML lokal) Nein (optional) + signal_outcomes, parameter_history, fehler_muster

6. WARUM DIESES DESIGN?

Kriterium Hybrid-Agent Warum
Läuft ohne Claude? ✓ Ja (100% funktional) Agent ist nicht Claude-abhängig, läuft 24/7 lokal
Intelligent? ✓ Ja (Decision-Trees + ML) Agent lernt, verbessert sich, nicht nur Regeln
Transparent? ✓ Ja (Z-4 erfüllt) Jede Entscheidung ist nachvollziehbar
Skalierbar? ✓ Ja (lokal + optional Cloud) Claude kommt später dazu, nicht notwendig
Budget? ✓ Niedrig-mittel Agent läuft billig lokal; Claude optional & sparsam
Wartbar? ✓ Ja Decision-Trees + ML sind Standard-Tech
